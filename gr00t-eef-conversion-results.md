# GR00T EEF Conversion — Results & Next Steps

**Date:** 2026-06-08
**Dataset:** `data/pick_place_bottle_v2_gr00t` → `data/pick_place_bottle_v2_gr00t_eef`
**Scope:** Bimanual (both arms)

---

## What Was Done

### Step 1 — Dataset conversion (`scripts/convert_to_eef_gr00t.py`)

All 202 episode parquets were rewritten with six new columns:

| New column | Dims | Source |
|---|---|---|
| `observation.left_eef_9d` | 9D | `observation.ee_pose[0:7]` → xyz + rot6d |
| `observation.right_eef_9d` | 9D | `observation.ee_pose[7:14]` → xyz + rot6d |
| `action.left_eef_9d` | 9D | Same as `observation.left_eef_9d` (kinesthetic) |
| `action.right_eef_9d` | 9D | Same as `observation.right_eef_9d` (kinesthetic) |
| `action.left_gripper` | 1D | `observation.state[6]` |
| `action.right_gripper` | 1D | `observation.state[13]` |

**EEF format:** `XYZ_ROT6D` — `[x, y, z, R[0,0], R[0,1], R[0,2], R[1,0], R[1,1], R[1,2]]`
where the 6D rotation is the first two rows of the rotation matrix (GR00T's native convention).

**Quaternion convention:** input `[x, y, z, w]` (xyzw, scipy default, matches Realman ROS2 output).

**Why absolute poses in the dataset?** GR00T's `state_action_processor.py` converts absolute →
relative at train time when `ActionRepresentation.RELATIVE` is set in the embodiment config.
No pre-subtraction needed.

All original columns (`observation.state`, `action`, `observation.ee_pose`, videos, etc.) are
kept untouched.

`meta/info.json` was updated with feature entries for all six new columns (dtype `float32`).

`meta/modality.json` was replaced with the EEF layout:

```json
{
  "state":  { "left_eef_9d": ..., "left_gripper": ..., "right_eef_9d": ..., "right_gripper": ... },
  "action": { "left_eef_9d": ..., "left_gripper": ..., "right_eef_9d": ..., "right_gripper": ... },
  "video":  { "top_camera": ..., "left_wrist": ..., "right_wrist": ... },
  "annotation": { "human.task_description": ... }
}
```

State grippers are sliced from `observation.state` (indices 6 and 13).
Action grippers are separate 1D columns (`action.left_gripper`, `action.right_gripper`).

## Next Steps

### Step 4 — Finetune (start from base model, not joint checkpoint)

```bash
cd Isaac-GR00T

CUDA_VISIBLE_DEVICES=0 uv run python gr00t/experiment/launch_finetune.py \
  --base-model-path nvidia/GR00T-N1.7-3B \
  --dataset-path ../data/pick_place_bottle_v2_gr00t_eef \
  --embodiment-tag NEW_EMBODIMENT \
  --modality-config-path examples/RealMan/realman_dual_arm_eef_config.py \
  --experiment-name gr00t_pick_place_bottle_eef \
  --output-dir ../outputs/train/gr00t_pick_place_eef \
  --max-steps 10000 \
  --global-batch-size 32 \
  --save-steps 1000 \
  --use-wandb \
  --wandb-project gr00t_pick_place
```

> **Important:** Do not continue from the joint-space checkpoint.
> The action head dimension and normalization stats are different.
> Always start from `nvidia/GR00T-N1.7-3B`.

### Step 5 — Open-loop evaluation (before robot deploy)

```bash
cd Isaac-GR00T

uv run python gr00t/eval/open_loop_eval.py \
  --model-path ../outputs/train/gr00t_pick_place_eef/gr00t_pick_place_bottle_eef/checkpoint-10000 \
  --dataset-path ../data/pick_place_bottle_v2_gr00t_eef \
  --embodiment-tag NEW_EMBODIMENT \
  --modality-config-path examples/RealMan/realman_dual_arm_eef_config.py \
  --traj-ids 0 50 101 150 201 \
  --action-horizon 16 \
  --modality-keys left_eef_9d right_eef_9d left_gripper right_gripper
```

Compare trajectory plots against the joint-space run. EEF MSE is in metres/radians,
not directly comparable to joint-space MAE.

## Known Limitations

- **Separate arm frames:** Left and right EEF are in each arm's own Realman base frame,
  not a shared world frame. This is handled correctly by treating them as independent
  modality keys; no cross-arm calibration needed for training.
- **No checkpoint transfer:** Joint → EEF is not supported; the action head shape differs.
- **Reset variability:** EEF start poses vary ~35 mm across episodes.
  Relative action representation (`RELATIVE`) mitigates drift vs absolute EEF targets.
